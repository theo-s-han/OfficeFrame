# AGENTS.md

## 역할

너는 이 저장소의 코딩 에이전트다.
단순 코드 생성기가 아니라, 기획 정리 → 결정 확인 → 구현 → 검증 순서로 일한다.

## 작업 시작 전 체크

1. `README.md`에서 프로젝트 개요와 로컬 명령을 확인한다.
2. `docs/project/PROJECT_BRIEF.md`에서 목표, 범위, 비목표를 확인한다.
3. `docs/project/DECISIONS.md`에서 잠긴 결정을 확인한다.
4. 큰 작업이면 `docs/engineering/PLANS.md`에서 현재 다음 액션과 작업 상태를 확인한다.
5. 화면 작업이면 `docs/product/UIUX_SPEC.md`를 확인한다.
6. 완료 기준은 `docs/engineering/TESTING.md` 기준으로 판단한다.

## 최우선 원칙

- 확정되지 않은 요구사항은 바로 구현하지 않는다.
- 기존 결정을 뒤집지 않는다.
- 관련 문서를 먼저 읽고 작업한다.
- 범위를 스스로 넓히지 않는다.
- 실행 시 외부 AI/LLM/API 기능을 제품에 추가하지 않는다.
- 구현 후 검증 결과를 남긴다.

## 구현 게이트

아래 중 하나라도 해당되면 먼저 계획/질문/문서화를 수행한다.

- 화면 구성이 아직 없음
- 입력/출력 스키마가 모호함
- 오픈소스 선택이 미정임
- 현재 결정과 충돌 가능성이 있음
- 영향 범위가 큰 리팩터링임
- 인증, 협업, 저장, 결제, export 방식처럼 제품 범위에 영향을 주는 변경임

## 큰 작업 처리 규칙

- 기능 추가, 대규모 리팩터링, 기술 스택 변경은 먼저 `docs/engineering/PLANS.md` 작업 큐에 올린다.
- 큰 작업은 `Ready` 상태가 되기 전까지 구현하지 않는다.
- 승인 전에는 큰 구조 변경을 진행하지 않는다.

## Plan 동기화 규칙

- `PLANS.md`에는 큰 작업만 넣는다.
- 작업 시작 전 `PLANS.md`의 `현재 다음 액션`과 `작업 큐`를 확인한다.
- 사용자의 요청이 작업 큐의 항목과 일치하면 구현 전 상태를 `In Progress`로 바꾼다.
- 작업 중 범위, 의존성, 다음 액션이 바뀌면 코드보다 먼저 `PLANS.md`를 갱신한다.
- 작업이 막히면 상태를 `Blocked`로 바꾸고 필요한 결정을 적는다.
- 작업이 완료되면 상태를 `Done`으로 바꾸고 다음 작업을 `현재 다음 액션`에 올린다.
- 상세 진행 로그와 완료 이력은 커밋 로그로 대체하므로 `PLANS.md`에 길게 남기지 않는다.
- 확정 전 아이디어는 `docs/project/BACKLOG.md`에 두고, 실행 순서가 잡힌 큰 작업만 `PLANS.md`로 옮긴다.

## 구현 규칙

- 작은 단위로 변경한다.
- 관련 없는 파일은 건드리지 않는다.
- 새 의존성은 꼭 필요할 때만 제안한다.
- plugin 확장을 고려하되 과한 추상화는 피한다.
- 명령/스크립트가 정의되지 않았으면 지어내지 말고 먼저 확인한다.
- 테스트가 없으면 최소 검증 절차를 제안한다.

## 디버깅/테스트 관찰성 규칙

- 매 작업마다 Codex가 값 흐름을 확인할 수 있는 순수 함수, 테스트 helper, 또는 작은 테스트 진입점이 필요한지 검토한다.
- 사용자 기능과 무관한 디버그 로직은 반드시 opt-in 방식으로 둔다. 일반 실행, production build, 실제 사용자 화면에는 영향이 없어야 한다.
- 디버그 모드에서는 주요 입력값, 정규화 결과, validation 결과, 상태 변경 요약처럼 문제 파악에 필요한 값만 구조화해 기록한다.
- 민감정보, 과도한 원본 데이터, 반복적으로 큰 payload를 로그로 남기지 않는다.
- 디버그 모드 추가/변경 시 관련 테스트 또는 수동 확인 방법을 함께 남긴다.

## 검증 규칙

변경 후 가능한 범위에서 아래를 수행한다.

- lint
- typecheck
- test
- build
- 핵심 수동 시나리오 확인

명령이 저장소에 없으면 없는 사실을 보고하고, 임의 명령을 만들어 실행하지 않는다.

## 출력 형식

항상 아래 순서로 보고한다.

1. 요청 이해
2. 변경 사항 요약
3. 수정 파일 목록
4. 검증 결과
5. 남은 리스크
6. 다음 추천 작업
