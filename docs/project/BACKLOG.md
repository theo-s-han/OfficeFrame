# BACKLOG.md

## 목적
후보 아이디어와 후순위 작업을 관리한다.
이 파일에 있다고 자동으로 구현하지 않는다.

## 운영 규칙
- backlog 항목은 `확정 아님` 상태로 취급한다.
- 구현 전에는 `PROJECT_BRIEF.md`와 `DECISIONS.md` 기준으로 재검토한다.
- 우선순위와 보류 이유를 함께 적는다.

## 항목 목록
| 아이템 | 사용자 가치 | 우선순위 | 상태 | 보류/제외 이유 |
|---|---|---|---|---|
| 간트 차트 MVP | 일정과 진행률을 문서용 이미지로 빠르게 생성 | High | Candidate | 실행 골격 다음 단계에서 구현 |
| PNG export | PPT/문서 붙여넣기에 바로 사용 | High | Candidate | preview 품질 확인 후 구현 |
| 마인드맵 | 아이디어 구조화 문서에 유용 | Medium | Candidate | 첫 도구 간트 이후 검토 |
| 조직도 | 조직/역할 설명 문서에 유용 | Medium | Candidate | 데이터 모델과 렌더러 검토 필요 |
| 플로우차트 | 프로세스 문서에 유용 | Medium | Candidate | 자유형 편집 대신 제한된 입력 방식 검토 필요 |
| 타임라인 | 일정/로드맵 문서에 유용 | Medium | Candidate | 간트 이후 시간축 계열 확장으로 검토 |
| WBS | 프로젝트 구조 문서에 유용 | Low | Candidate | 간트/조직도 기반이 잡힌 뒤 검토 |
| ERD/시퀀스 다이어그램 | 개발 문서 가치가 높음 | Low | Candidate | 초기 실무형 범위보다 무거움 |
| 인증/협업/클라우드 저장 | 팀 단위 사용성 개선 | Low | Excluded | 초기 제품 제약상 제외 |
| 고급 drag/drop 편집 | 세밀한 수정 가능 | Low | Deferred | 구현 비용이 높고 MVP 핵심이 아님 |

## 검토 체크리스트
- 이 기능이 핵심 문제를 직접 해결하는가?
- MVP 성공률을 올리는가?
- 기존 구조와 충돌하지 않는가?
- 프론트엔드만으로 시작 가능한가?
- 검증 가능한 완료 기준을 만들 수 있는가?
